# wysJSON 项目记忆体

这份文档不是产品宣传，也不是泛化 README。
它的目标是：换电脑、换会话、换模型之后，能快速恢复当前项目语境，继续高效配合做 vibe coding。

## 1. 当前项目是什么

`wysJSON` 目前是一个单文件原型：`index.html`。

它不是完整 VS Code 扩展，而是一个可直接在浏览器打开的 JSON 表格编辑器，核心目标是：

- 用类似 Excel 的交互编辑 JSON
- 支持 object / array / primitive 混合嵌套
- 尽量所有操作都通过 UI 完成，而不是手改 JSON 文本
- 为后续封装成 VS Code Webview 保留结构基础

当前 workspace 里最重要的文件：

- `index.html`
  现阶段全部功能都在这里，包含 HTML、CSS、状态、渲染、编辑、右键菜单、递归表格逻辑。
- `wysLog/README.md`
  这份“项目记忆体”。
- `jsonDemo/`
  放样例 JSON，可用于人工测试。

## 2. 当前技术现实

这是一个纯前端原型，没有构建工具，没有打包步骤，没有依赖安装。

实际运行方式：

- 直接打开 `index.html`
- 或用本地静态服务器打开 `index.html`

不要把它当成 Node 项目来理解。

注意：

- `node --check index.html` 对 `.html` 文件没有意义，报错不代表脚本语法坏了。
- 真正有效的验证方式是：浏览器打开页面后做交互验证，或者用 Playwright 检查 DOM/菜单/状态。

## 3. 当前代码结构

整体是一个全局 `App` 对象，集中管理状态和行为。

关键职责分布：

- 初始化与绑定：`init()`、`bindEvents()`
- 顶层渲染：`render()`
- 递归渲染：`createTable()`、`createSimpleView()`、`renderCell()`
- 路径访问：`parsePath()`、`getValueAtPath()`、`setValueAtPath()`
- 编辑与类型推断：`beginEdit()`、`commitEdit()`、`parseValue()`
- 结构操作：`addRow()`、`deleteRow()`、`addColumn()`、`deleteColumn()`
- 右键菜单：`handleTableContextMenu()`、`showContextMenu()`、`handleContextMenuAction()`
- 批量编辑：选区复制、粘贴、拖拽填充
- 历史记录：`pushUndo()`、`undo()`、`redo()`

判断行为时，优先从这几个入口读，不要先从 CSS 或零散 DOM 开始猜。

## 4. 当前渲染规则

这是本项目最重要的记忆点之一。

### 4.1 object 渲染

- object 默认渲染成单行表格
- key 是列头
- value 是 cell
- 嵌套 object / array 在 cell 内递归渲染
- 空 object 不能再递归套自己；应显示为空对象占位，并提供 `+ 添加列` 入口

### 4.2 array 渲染

array 现在有两种模式：

1. `纯 object 数组`
  走列式表格模式。
  也就是数组每个元素是一行，对象字段汇总成列。

2. `混合数组` 或 `非纯 object 数组`
  走逐行节点模式。
  每一行只有“行号 + 一个值节点”。
  这个值节点可以是：
  - primitive
  - object
  - array
  - null
  并且可以继续无限递归。

当前判定规则是：

- 只有当数组中每一个元素都是 plain object 时，才走列式表格模式
- 只要混入 string / number / boolean / null / array，或者结构不统一，就走逐行节点模式
- mixed array 点“添加项”时，不要猜测共享结构；默认新增 `null`（或在 `nullAsString` 开启时新增空字符串）

这条规则是为了解决类似下面这种数据：

```json
[{"rrr":"JavaScript"},"React","Node.js"]
```

这种混合 array 不能再按 object 表格硬套列头，否则 primitive 元素会显示错误，也无法递归编辑。

## 5. 当前交互规则

### 5.1 单元格编辑

- primitive cell 支持直接编辑
- 输入内容经过 `parseValue()` 自动推断类型
- 支持：`null`、`true`、`false`、number、内联 JSON（`{}` / `[]`）
- 推断失败则按字符串处理

### 5.2 行号行为

- array 行号左键：删除该行
- object 的单行行号左键：将该对象设为 `null` 或空字符串（取决于 `nullAsString`）
- array 行号右键：弹出“改变类型”菜单
- simple view 的 array 行号右键也已经支持相同行为

### 5.3 cell 右键菜单

cell 右键菜单现在支持把当前节点改成：

- `Array`
- `Object`
- `单值`
- `String`
- `Number`
- `Boolean`
- `Null`

如果当前节点本身是 object / array，右键菜单还应直接提供：

- object: `➕ 向当前对象添加字段`
- array: `➕ 向当前数组添加项`

注意：这里改的是当前 cell 对应路径上的值，不是整个父表格的结构类型。

### 5.4 左上角 corner 菜单

表格左上角的 corner 菜单目前用于“当前表/选区级别”的结构操作：

- 删除选区节点（置为 `null`）
- 改变节点类型为 `Array`
- 改变节点类型为 `Object`
- 改变节点类型为 `单值`

这是“节点级别”的批量操作，不是 cell 文本样式操作。

### 5.5 行号 / 嵌套节点新增路径

- 纯 object 数组：底部 `+ 添加行` 继续按对象模板新增
- mixed array：底部 `+ 添加项` 默认新增 `null` 或空字符串，不猜共享结构
- array 行号右键：支持 `在当前行前插入`、`在当前行后插入`、`删除当前行`
- 空 object：必须显示占位态 `空对象`，并提供 `+ 添加列`
- 嵌套 object / array：除了底部按钮，现在也可以通过右键菜单直接添加子字段 / 子项

### 5.6 列头字段操作

- object / 纯 object 数组的列头右键：支持 `在当前列前插入`、`在当前列后插入`、`删除列`
- 插入列后应自动进入列名编辑态
- 对纯 object 数组插入列时，所有对象行都要同步插入该字段，并保持字段顺序一致

### 5.7 结构变更后的焦点规则

- `+ 添加行` / `+ 添加项` 后，焦点应自动落到新行对应的第一个可选 cell
- `在当前行前插入` / `在当前行后插入` 后，焦点应自动落到插入的新行
- 删除行后，焦点应尽量落到相邻保留下来的行
- 插入列后，焦点继续交给列名编辑框，不要被 cell 选中逻辑打断
- 删除列后，焦点应尽量留在同一行的相邻列；如果当前行不可用，再退到该对象/表格中的稳定相邻列

### 5.8 单值规则

这是一个被反复确认过的规则，不要改错。

- `['a', 'b'] -> 'a'`
- `{ "a": 1, "b": 2 } -> 1`
- 如果嵌套更深，递归取“第一个元素 / 第一个字段”的值

也就是说“改成单值”不是简单包一层，而是递归收敛到首值。

### 5.9 object / array 的转换规则

- `object -> array`
  不丢数据，变成 `[原object]`

- `array -> object`
  只保留第一项
  - 第一项如果本身是 object，直接使用
  - 第一项如果是 primitive，包成 `{ value: 第一项 }`

- `primitive -> array`
  变成 `[value]`

- `primitive -> object`
  变成 `{ value: value }`

## 7. 现在仍然要小心的点

下面这些是继续改动时最容易出问题的地方。

### 7.1 这是单文件应用

`index.html` 同时承载结构、样式、状态和逻辑。

这意味着：

- 小补丁容易插错函数位置
- 改动一个渲染分支，可能影响所有递归层级
- 运行时问题常常不是语义问题，而是“补丁插错位置导致脚本结构被污染”

因此每次编辑后都要马上做一次浏览器级验证。

### 7.2 浏览器标签页容易旧缓存

之前多次出现“文件已经改了，但当前测试页还是旧逻辑”的情况。

继续协作时优先做：

- 硬刷新页面
- 或重新打开 `file:///.../index.html`
- 不要默认相信旧 tab 的行为

### 7.3 节点类型和 cell 类型必须区分

这个需求是后来明确加上的。

要始终区分：

- “当前节点值的类型转换”
- “整个表/选区节点的结构转换”

不要再把 cell 菜单和 corner 菜单混成一套语义。

### 7.4 混合数组不能退化成 object 表格

只要 array 不是纯 object 数组，就必须逐行递归。

不要为了“表格更整齐”把混合数组重新聚合成统一列结构，否则会再次破坏递归和类型独立性。

## 8. 推荐验证方式

以后接手时，优先用这几组数据手测：

### 8.1 基础样例

使用内置 `loadSample()` 数据，验证：

- object 表格渲染
- primitive array
- nested object
- nested object array

### 8.2 混合数组样例

```json
[{"rrr":"JavaScript"},"React","Node.js"]
```

重点验证：

- 第 1 行 object 能递归显示
- 第 2 / 3 行 primitive 正常显示
- 行号右键可改类型
- cell 右键可改类型

### 8.3 单值转换样例

```json
{
  "a": ["x", "y"],
  "b": {"k": 1, "m": 2}
}
```

重点验证：

- `a -> 单值` 得到 `"x"`
- `b -> 单值` 得到 `1`

### 8.4 object / array 转换样例

```json
{
  "x": {"name": "A"},
  "y": [1, 2],
  "z": "hello"
}
```

重点验证：

- `x -> array` 变 `[ {"name":"A"} ]`
- `y -> object` 变 `{ "value": 1 }` 或首项 object
- `z -> object` 变 `{ "value": "hello" }`
- `z -> array` 变 `["hello"]`

## 9. 如果未来继续和大模型协作，建议怎么开场

推荐把下面这类信息一次性给模型：

1. 当前目标
  例：修 mixed array 下的某个右键行为

2. 当前关键规则
  例：
  - 纯 object array 才走列式表格
  - 混合 array 必须逐行递归
  - cell 菜单只改当前值
  - corner 菜单是节点级批量操作

3. 需要先确认的点
  如果一个交互语义有歧义，先让模型复述理解，再改

4. 明确要求
  - 先定位根因
  - 再做最小改动
  - 改后立即做浏览器验证

推荐提示词风格：

```text
先读 README，把它当项目记忆体。
先确认你对当前规则的理解，再改。
优先修根因，不要只改表层显示。
改完后用浏览器实际验证，不要只看代码。
```

## 10. 下一阶段可继续做什么

如果后续继续开发，优先级建议如下：

1. 完善“全 UI 操控所有节点”
  包括混合 array / nested object / nested array 的增删改和类型切换边角行为。

2. 补齐结构操作菜单
  比如：
  - 新增 sibling 节点
  - 在 object 中新增字段
  - 在 array 指定位置插入新项
  - 复制路径 / 复制 JSON

3. 把 `index.html` 逐步拆分
  至少先把渲染、路径操作、菜单逻辑分段整理，降低补丁风险。

4. 再考虑 VS Code Webview 封装
  当前先把编辑器原型做扎实，扩展封装放后面更稳。

## 11. 当前共识

以下是本项目目前已经反复确认过的共识，后续默认遵守：

- 优先做能直接工作的 UI 编辑体验
- 先修根因，再修表象
- 对有歧义的交互，先确认再改
- 每次改完优先浏览器验证，不迷信静态阅读
- 混合 array 必须支持无限递归
- 不要强迫 array 所有元素同构
- 节点类型与 cell 类型要明确区分
